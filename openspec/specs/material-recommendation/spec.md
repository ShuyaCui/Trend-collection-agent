# Spec: Material Recommendation

## ADDED Requirements

### Requirement: Load and format material library for LLM context
The system SHALL load all elements from `material_library/color.json`, `material_library/texture.json`, and `material_library/decoration.json`, and format them into a compact text representation suitable for LLM system prompt injection.

#### Scenario: All three dimension files are loaded
- **WHEN** the recommender agent starts processing a query
- **THEN** the system SHALL load elements from all three JSON files (color, texture, decoration) and include all of them in the LLM prompt context

#### Scenario: Each element preserves key attributes
- **WHEN** an element is formatted for the prompt
- **THEN** the formatted text SHALL include the element's `id`, `name`, `dimension`, `visual_keywords` (up to 10), `signals` (up to 10), and `typical_use`. The `source_report` and `source_heading` fields SHALL NOT be included in the prompt text (they are used for post-hoc traceability lookup only)

#### Scenario: Material library files are missing
- **WHEN** any of the three JSON files does not exist or is malformed
- **THEN** the system SHALL raise an informative error indicating which file is missing or invalid

### Requirement: Structured recommendation output
The system SHALL produce recommendations as a Pydantic `RecommendationResult` model via `model.with_structured_output()`, containing three lists (`colors`, `textures`, `decorations`) of `ElementRecommendation` objects.

#### Scenario: Successful recommendation with default count
- **WHEN** a user submits a design query without specifying a count (e.g., "设计一款以酸奶为概念的沐浴露")
- **THEN** the system SHALL return exactly 5 `ElementRecommendation` items per dimension (colors, textures, decorations), or all available elements if fewer than 5 exist in that dimension

#### Scenario: User-specified recommendation count
- **WHEN** a user specifies a count in their query (e.g., "每个维度推荐3个")
- **THEN** the system SHALL return exactly that many `ElementRecommendation` items per dimension

#### Scenario: Each recommendation contains required fields
- **WHEN** an `ElementRecommendation` is produced
- **THEN** it SHALL contain: `element_id` (matching the element's `id` in the library), `element_name`, `element_name_en`, `dimension`, `relevance_score` (float 0-1), `reasoning` (1-2 sentence justification), `source_reports` (list of report IDs), and `source_heading`

### Requirement: Concept analysis in output
The system SHALL include a `concept_analysis` field in the `RecommendationResult` that explains how the LLM decomposed the user's design concept into relevant aesthetic attributes, moods, and associations.

#### Scenario: Concept analysis reflects query decomposition
- **WHEN** a user submits "设计一款以酸奶为概念的沐浴露"
- **THEN** the `concept_analysis` SHALL describe the conceptual associations derived from "酸奶" (e.g., milky white, layered, creamy, fermented, fruity) and how they map to design dimensions

### Requirement: Source traceability in recommendations
Each `ElementRecommendation` SHALL include traceability to the original trend report. The `source_reports` and `source_heading` fields SHALL be populated by a **post-hoc lookup** step: after the LLM produces recommendations with `element_id`, the system SHALL look up the corresponding element in the material library JSON and copy its `source_report` and `source_heading` values. These fields SHALL NOT be generated by the LLM.

#### Scenario: Source fields are populated via lookup
- **WHEN** the LLM recommends element "分层乳带" (id: `25fccecd-decoration-c7a23ed0-8ce5b0`)
- **THEN** the system SHALL look up that element's `source_report` and `source_heading` from `decoration.json` and populate `source_reports` with `["f7b105e7-cb9d-4dfa-bbd8-031590936687/report.md"]` and `source_heading` with "Blueglass 酸奶分层饮品视觉参考"

#### Scenario: Elements with multiple source reports
- **WHEN** an element's `source_report` field contains multiple report paths separated by " + "
- **THEN** the `source_reports` list SHALL contain all unique report paths, deduplicated

### Requirement: Multi-turn interactive refinement
The recommender agent SHALL support multi-turn conversations where users refine recommendations based on previous results. The agent SHALL retain full conversation history via LangGraph's `MessagesState` and adjust subsequent recommendations accordingly.

#### Scenario: User requests replacement recommendations
- **WHEN** a user says "换一批" after receiving initial recommendations
- **THEN** the system SHALL produce a new set of recommendations that does NOT repeat elements from the previous response (where possible given library size)

#### Scenario: User adds constraints
- **WHEN** a user says "更偏清新自然的风格" after initial recommendations
- **THEN** the system SHALL produce new recommendations weighted toward the "清新自然" aesthetic, while still maintaining diversity across the dimension

#### Scenario: User adjusts scope
- **WHEN** a user says "去掉饮品类的元素" after initial recommendations
- **THEN** the system SHALL exclude elements with `product_category: "饮品"` from subsequent recommendations

#### Scenario: Conversation history is preserved
- **WHEN** the user sends a third message in a conversation thread
- **THEN** the LLM SHALL have access to all previous user messages and assistant responses to maintain context

### Requirement: LangGraph agent registration
The recommender agent SHALL be registered as a separate graph entry in `langgraph.json` with the key `material_recommender`.

#### Scenario: Agent is discoverable via LangGraph API
- **WHEN** the LangGraph server starts
- **THEN** the `material_recommender` graph SHALL be available at the `material_recommender` endpoint and accept `MessagesState`-compatible input

### Requirement: Cross-category creative association
The system SHALL NOT filter elements by `product_category`. All elements from all product categories SHALL be included in the recommendation candidate pool, encouraging the LLM to make cross-category creative associations (e.g., beverage elements inspiring body care design).

#### Scenario: Beverage elements recommended for body care query
- **WHEN** a user asks for shower gel design recommendations
- **THEN** the system SHALL consider elements with `product_category: "饮品"` as valid candidates if they are conceptually relevant (e.g., "分层乳带" from beverages for a yogurt-concept shower gel)

#### Scenario: No product category filtering in prompt
- **WHEN** the material library is formatted for the LLM prompt
- **THEN** all elements from all product categories SHALL be included without any filtering or separation by category

### Requirement: Recommendation prompt template
The system SHALL define recommendation prompt templates in `prompts.py`, including a system prompt that contains: recommendation rules, the full formatted material library, and output format instructions.

#### Scenario: Prompt includes recommendation rules
- **WHEN** the recommendation prompt is constructed
- **THEN** it SHALL instruct the LLM to: prioritize conceptual relevance, maintain diversity across recommendations, provide reasoning for each choice, and use source fields from the library data for traceability

#### Scenario: Prompt includes full material library
- **WHEN** the recommendation prompt is constructed
- **THEN** the system prompt SHALL contain all elements from all three dimensions, formatted as compact text

### Requirement: Azure OpenAI authentication
The recommender agent SHALL use the same `GenAIToken`-based Azure AD authentication as existing agents, with model initialized via `init_chat_model("azure_openai:...")`.

#### Scenario: Auth follows existing pattern
- **WHEN** the recommender LLM is initialized
- **THEN** it SHALL use `GenAIToken` for bearer token auth with auto-refresh, reading `AZURE_OPENAI_ENDPOINT`, `AZURE_OPENAI_DEPLOYMENT`, and `AZURE_OPENAI_API_VERSION` from environment variables
