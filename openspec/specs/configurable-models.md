# Spec: Runtime-Configurable LLM Models

## Summary

All LLM models are currently instantiated at **module load time** with hardcoded model identifiers. This feature makes each node's model configurable via LangGraph's standard `config["configurable"]` mechanism, allowing operators and users to select models at **invocation time** without modifying source code.

---

## Goal

When invoking any LangGraph agent graph, users can pass `config={"configurable": {...}}` to specify which model each role uses, instead of relying on hardcoded defaults in source code.

---

## Scope

Six model roles are covered (one added vs. initial draft based on `summarization_model`):

| Configurable key      | Current default        | File                                                                              |
| --------------------- | ---------------------- | --------------------------------------------------------------------------------- |
| `scope_model`         | `azure_openai:gpt-4.1` | `research_agent_scope.py` (clarify + brief nodes)                                 |
| `research_model`      | `azure_openai:gpt-4.1` | `research_agent.py` (`llm_call` node)                                             |
| `summarization_model` | `azure_openai:gpt-4.1` | `utils.py` via `research_agent.py` runtime config (webpage content summarization) |
| `compress_model`      | `azure_openai:gpt-4.1` | `research_agent.py` (`compress_research` node)                                    |
| `supervisor_model`    | `azure_openai:gpt-5.3` | `multi_agent_supervisor.py`                                                       |
| `writer_model`        | `azure_openai:gpt-5.3` | `research_agent_full.py`                                                          |

Implementation approach:

- Model objects are initialized **inside each node function** (lazy init), reading the model name from `config["configurable"]` with a fallback to the existing default.
- `model_with_tools` binding (`bind_tools`) is also done inside the node on each call.
- Tool-side webpage summarization reads runtime config through a context-local bridge so `summarization_model` and `research_model` overrides apply during Tavily result processing.
- **Convention**: the model name string is also used as the Azure deployment name (e.g., `"azure_openai:gpt-4.1"` → deployment `"gpt-4.1"`). `AZURE_OPENAI_DEPLOYMENT` env var is retired from model init.
- All changes are made in the `%%writefile` cells of the corresponding notebooks. `src/` files are regenerated by running those cells.

---

## Non-goals

- Making Azure auth credentials (endpoint, API key, API version) configurable — these remain in env vars.
- Making `temperature` or `max_tokens` configurable per invocation (this version).
- Changing Tavily or MCP tool configurations.
- Adding a UI or API layer to expose model selection.
- Supporting non-Azure providers (current deployment is Azure-only).

---

## Constraints

1. **Notebooks are source of truth.** All changes must be in `%%writefile` cells in the relevant notebooks; never edit `src/` directly.
2. **Backward compatible.** Calling any graph without a `config` argument must produce identical behavior to the current version (defaults cover all roles).
3. **No model caching.** `init_chat_model(...)` is called fresh on each node invocation; no `lru_cache` or model registry is introduced.
4. **Model name = deployment name.** The string passed as model (e.g., `"gpt-4.1"`) is extracted and used as `azure_deployment`. The `AZURE_OPENAI_DEPLOYMENT` env var is no longer used in `init_chat_model` calls.

---

## Acceptance Criteria

1. Each of the four graph entry points (`scope_research`, `researcher_agent`, `supervisor_agent`, `research_agent_full`) accepts `config={"configurable": {<role_key>: "<model>"}}` and uses the specified model in the corresponding node.
2. Invoking any graph without a `config` argument produces the same results as before this change (defaults are unchanged).
3. All six model roles have no hardcoded model names at module level.
4. Passing a single `config` to `research_agent_full` propagates configurable values through all subgraphs (`supervisor_agent` → `researcher_agent`).
5. `ruff check src/` passes with no new errors after notebook cells are re-run.

---

## Validation Plan

1. Call `scope_research` with `config={"configurable": {"scope_model": "azure_openai:gpt-4.1"}}` — verify no error and model is used.
2. Call `researcher_agent` with `config={"configurable": {"research_model": "azure_openai:gpt-4.1", "compress_model": "azure_openai:gpt-4.1"}}` — verify correct models are picked up.
3. Call `supervisor_agent` with `config={"configurable": {"supervisor_model": "azure_openai:gpt-4.1"}}` — verify supervisor node uses the overridden model.
4. Call `research_agent_full` with `config={"configurable": {"writer_model": "azure_openai:gpt-4.1"}}` — verify writer node uses it and subgraph models default correctly.
5. Call all graphs with no `config` and confirm behavior matches pre-change baseline.
6. Run `ruff check src/` and confirm no new lint errors.

---

## Risks / Open Questions

**R1. Azure deployment name extraction** _(Decision: model name = deployment name)_  
The model string format is `"azure_openai:<deployment>"`. When building `init_chat_model`, the deployment name is extracted by splitting on `:` (e.g., `"azure_openai:gpt-4.1"` → deployment `"gpt-4.1"`).  
→ Users must ensure the model string they pass matches an actual deployment in their Azure resource.

**R2. Config propagation through subgraphs**  
LangGraph propagates `configurable` config to subgraphs automatically when using `.compile()`. This needs to be verified for the `full_agent` → `supervisor_agent` → `researcher_agent` chain before the feature is considered done.

**R3. `summarization_model` usage**  
`summarization_model` applies to webpage summarization in `utils.py`. If it is omitted at runtime, the tool-side helper falls back to `research_model`, then to the default summarization model.
